Document Processing Service

GT Alumni Collaboration  ·  Spring 2026
Python PyTorch HuggingFace Transformers LayoutLMv3 MLflow Docker AWS

When I started at Georgia Tech in Spring 2026, I wanted a project that would force me to actually learn machine learning, not just read about it. This was it.

The idea was practical: build a service that extracts structured data from documents like receipts automatically. The kind of thing that saves a business hours of manual data entry, but flexible enough that anyone could use it on their own documents. We designed it with commercial intent, something that could realistically be sold as a service.

The core is a fine-tuned LayoutLMv3, a 133M parameter multimodal transformer that understands documents by combining the text, the layout, and the visual appearance of the page together. We trained it on the SROIE receipt dataset to extract four fields: Vendor, Address, Total, and Date, reaching an F1 score of 0.83. Getting there meant running a lot of experiments: tweaking hyperparameters, adjusting label schemes, comparing runs. MLflow tracked everything, which made it possible to actually learn from the iteration instead of losing track of what changed between runs.

On top of the extraction layer, we built a RAG-powered query interface with a semantic router that classifies incoming questions and dispatches them to the right path: structured SQL generation, pgvector similarity search, or a hybrid path that uses Claude Haiku via AWS Bedrock for query generation and answer synthesis.

The whole thing is deployed on AWS: FastAPI for the inference server, EC2 for compute, SQS for the job queue, Postgres and RDS for storage, and Docker throughout. OCR runs as its own service using Tesseract.

This was the first project where I trained and compared many models in a real experimental loop.

Protein Folding Optimization

VT Capstone  ·  Spring 2025
Python AMBER Molecular Dynamics

When AlphaFold won the Nobel Prize in Chemistry in 2024, it made protein folding feel like a real and tractable problem for the first time. That was part of what drew me to this project. We were not using deep learning like AlphaFold, but we were working on the same underlying challenge: figure out how a protein folds by finding the lowest-energy configuration it can settle into.

The difficulty is that the energy landscape is rugged. There are countless local minima, configurations that look stable but are not the true global minimum, and most optimizers get trapped in them. Our task was to benchmark five methods against this kind of landscape: Newton's Method, Conjugate Gradient, Differential Evolution, Monte Carlo, and Simulated Annealing.

Simulated annealing came out on top. The reason it works well here is that it can escape local minima by occasionally accepting worse solutions, similar to how slowly cooling metal settles into a more stable crystalline structure. We pushed it further by designing multi-cycle annealing protocols with controlled temperature peaks and cooling phases, which improved on single-cycle baselines.

All simulations ran on the AMBER molecular dynamics framework with implicit solvent modeling, applied to a 20-residue miniprotein called Protein T.

Most of my undergrad projects were pieces of something larger, useful in context but not really on their own. This felt different. We had a real problem, a set of tools, and a question with an actual answer at the end of it.
